# Задача 2 — свой HTTP-сервер на чистом Node

Вчера стучались в чужой сервер. Теперь поднимаешь свой — на голом модуле `http`, без фреймворков. Он держит справочник сотрудников в памяти и отвечает на запросы тем же ресурсом `/users`, что и API из первой задачи. В конце перенаправишь свой клиент из задачи 1 на этот сервер и убедишься, что он работает точно так же.

## Подготовка проекта

Собери новый проект по шагам с лекции:

1. `mkdir http-server && cd http-server && git init`
2. `echo "lts/*" > .nvmrc && nvm use`
3. `npm init -y`
4. `.gitignore` под Node (`node_modules`, `.env`, `*.log`)
5. `npm install -D prettier eslint`, потом `npx eslint --init`
6. `.prettierrc` + `.prettierignore`
7. Точка входа `server.js`, скрипт `"start": "node server.js"` в `package.json`

Модули — `commonjs`.

## Что за сервер

`http.createServer` принимает функцию, которая вызывается на каждый входящий запрос — в ней читаешь `req` и формируешь `res`. Маршрутизацию делаешь вручную: смотришь на `req.method` и `req.url` и решаешь, что ответить. Список сотрудников держишь в обычном массиве внутри процесса.

Сервер слушает порт 3000. Все ответы — с заголовком `Content-Type: application/json`, тело собираешь через `JSON.stringify`.

## Маршруты

- `GET /users` — вернуть весь массив, статус 200;
- `GET /users/<id>` — вернуть одного сотрудника по id (id вытащи из `req.url`), статус 200; если такого нет — 404 с телом-ошибкой;
- `POST /users` — прочитать тело запроса, создать сотрудника с новым уникальным id, вернуть 201 и созданный объект;
- известный путь, но неподходящий метод (например `DELETE /users`) — 405;
- всё остальное — 404.

## Чтение тела POST-запроса

В сыром Node тело приходит по кусочкам, поэтому его надо собрать. Это тот самый неочевидный приём — вот готовая схема, вставь её и пользуйся, логику маршрутов пиши сама:

```js
function readBody(req) {
  return new Promise((resolve, reject) => {
    let data = '';
    req.on('data', (chunk) => (data += chunk));
    req.on('end', () => resolve(data));
    req.on('error', reject);
  });
}
```

Дальше внутри обработчика: `const raw = await readBody(req);` и затем `JSON.parse(raw)`. Чтобы можно было делать `await`, оберни колбэк `createServer` в `async`.

## Входные данные

Стартовый массив сотрудников (захардкодь в `server.js` как начальное состояние):

const initial = [

## Проверка связки с задачей 1

Когда сервер поднят: возьми свой fetch-клиент из задачи 1 и поменяй в нём `BASE_URL` на `http://localhost:3000`. Ничего больше не трогая, запусти те же функции — они должны отработать против твоего сервера так же, как против публичного API. Это и есть смысл общего протокола: клиент не знает и не должен знать, чей сервер на том конце.

## Ожидаемый результат

Сервер поднимается на `http://localhost:3000` и на `GET /users` отдаёт статус 200 и массив из трёх сотрудников. `GET /users/2` возвращает 200 и Бориса; `GET /users/99` — 404 с сообщением об ошибке. `POST /users` с телом-json создаёт четвёртого сотрудника, возвращает 201 и его же с присвоенным id; следующий `GET /users` уже показывает четверых. Запрос `DELETE /users` отвечает 405, потому что путь известен, а метод не поддержан. Любой неизвестный путь даёт 404. Перенаправленный fetch-клиент из задачи 1 (с `BASE_URL = http://localhost:3000`) успешно получает список, одного сотрудника и создаёт нового — теми же функциями, что ходили в публичный API.
